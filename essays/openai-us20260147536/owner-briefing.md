# Owner Briefing (발행자 브리핑)

- **특허 번호**: US 2026/0147536 A1 (출원번호 19/397,773)
- **제목**: Alignment in Hardware Accelerators
- **상태**: 출원 (공개 공보; 등록 전, 심사 계속 중. 이 공보에서 청구항 11-19은 "(canceled)" 로 표기되어 있다)
- **출원일**: 2025-11-21 (가출원 63/724,159: 2024-11-22)
- **발명자**: Burak Erbagci, Cagla Cakir, Alexander Almela Conklin, Tracey DellaRova, Jean-Didier Allegrucci
- **출원인/양수인**: OpenAI Opco, LLC

**한 줄 요약**: OpenAI 명의로 공개된 이 출원은 저정밀 부동소수점(FP4/FP6/FP8) 행렬 곱셈을 메모리 안에서 수행하되, 곱들의 가수(mantissa)를 한 번만 정렬해 정수 덧셈기 트리로 누산하는 컴퓨트-인-메모리(compute-in-memory) 회로를 청구한다.

## ① 과거 기술의 과제/문제점

AI 학습과 추론은 압축된 숫자 형식으로 수행되는 대량의 행렬-벡터 곱셈에 의존한다 `[0002]` `[0004]`. 기존 파이프라인은 활성값(activation)과 가중치(weight)를 서로 다른 숫자 도메인(고정밀 부동소수점과 저정밀 정수) 사이에서 반복 변환하며, 변환 단계마다 스케일링, 반올림, 포화(saturation) 논리가 추가되어 면적, 전력, 지연이 늘고 층을 거치며 양자화 오차가 누적될 수 있다 `[0005]`. 또한 8비트 이하의 새 부동소수점 변형이 계속 등장하는데, 형식별로 고정된 변환 파이프라인은 이를 따라가기 어렵다 `[0005]`.

**근거 (verbatim):**
- `[0005]`: "Each conversion step requires additional scaling, rounding, and saturation logic, which adds area, power, and latency to already dense compute fabrics, and can introduce cumulative quantization errors that degrade numerical behavior over many layers."
  (변환 단계마다 스케일링, 반올림, 포화 논리가 추가되어 면적, 전력, 지연이 늘고, 여러 층에 걸쳐 양자화 오차가 누적될 수 있다는 뜻)
- `[0005]`: "These operations can involve transforming activations and weights back and forth between different numeric domains"
  (이 연산들은 활성값과 가중치를 서로 다른 숫자 도메인 사이에서 반복 변환하는 과정을 수반할 수 있다는 뜻)

## ② 기존 기술의 해결 방향과 그 한계

업계는 GPU, TPU, FPGA, ASIC 같은 전용 가속기로 이 부담을 덜어 왔다 `[0003]`. 그러나 통상의 가속기 구조는 메모리에 저장된 데이터를 연산 코어로 옮겨 와 계산하는 방식이라 데이터 이동 자체가 병목(폰 노이만 병목)이 되고 `[0133]`, 형식별 전용 데이터패스와 변환 파이프라인은 새 형식이 나올 때마다 재설계를 요구한다 `[0005]`. 명세서는 이 두 한계를 출발점으로 삼는다.

**근거 (verbatim):**
- `[0133]`: "This architecture is beneficial for accelerating AI workloads like deep learning by reducing the von Neumann bottleneck."
  (메모리 내 연산 구조가 폰 노이만 병목을 줄여 딥러닝 같은 AI 워크로드 가속에 유리하다는 뜻; 종래 구조의 한계를 뒤집어 말한 문장)
- `[0005]`: "fixed conversion pipelines and format-specific data paths become increasingly difficult to adapt, limiting the flexibility of the hardware to support evolving models and deployment scenarios"
  (고정된 변환 파이프라인과 형식별 데이터패스는 진화하는 모델과 배치 시나리오를 지원하기 점점 어려워진다는 뜻)

## ③ 이 특허의 독창적 해법 (핵심 청구항 중심)

핵심 독립항은 청구항 1, 20, 29이다. 출원 건이므로 확정된(locked) 범위는 없고, 아래는 모두 청구 중(sought)인 범위다.

청구항 1이 실제로 요구하는 것은 세 요소다:

- (a) 부동소수점 형식의 활성값과 가중치의 원시 곱(primitive product)을 만드는 컬럼 셀(column cell) `[0011]`
- (b) 그 곱들의 가수 비트를 시프트로 정렬하는 기능 블록 `[0011]`
- (c) 정렬된 가수를 더해 정수 형식의 누산값을 내는 덧셈기 트리(adder tree) `[0011]`
- 독립항 20은 같은 삼요소에 시프트 계산·선택 디코딩 유닛을 내장하고, 독립항 29는 시프트 레지스터를 내장한다
- 종속항은 시프트 계산·디코딩 유닛(청구항 2), 크기 규칙이 정해진 시프트 레지스터(청구항 3), 로그 구조 멀티플렉서 트리(청구항 6-9) 등을 추가한다
- 주의: 명세서에만 있는 것: 형식 유연성(모드 디코딩 유닛 `[0006]`), 이중 뱅크 병렬 쓰기 `[0012]`, 스캔 테스트 모드 `[0013]`. 청구항에 없음. 명세서 선호(open)와 청구 범위(sought)를 섞지 말 것. 청구항 11-19은 이 공보에서 취소로 표기되어 있다

**근거 (verbatim):**
- `[0011]`: "a functional block configured to align mantissa bits of primitive products generated by column cells in the computing unit by shifting the mantissa bits"
  (컴퓨팅 유닛 안 컬럼 셀들이 만든 원시 곱들의 가수 비트를 시프트 방식으로 정렬하는 기능 블록이라는 뜻; 청구항 1의 (b) 요소와 같은 문언)
- `[0011]`: "an adder tree configured to output an accumulation value of the primitive products in an integer format by adding the shifted mantissa bits"
  (시프트된 가수 비트를 더해 원시 곱들의 누산값을 정수 형식으로 출력하는 덧셈기 트리라는 뜻; 청구항 1의 (c) 요소와 같은 문언)

## ④ 기대 효과

명세서가 주장하는 효과로 한정하면 다음과 같다. 저정밀 형식 사용은 정확도 손실을 최소화하면서 연산 속도와 메모리 사용을 크게 개선한다 `[0131]`. 파이프라인 구조로 매 사이클 벡터 단위의 내적 결과를 반환한다 `[0147]`. 컬럼당 INT35/INT5(35비트 정수 합 + 5비트 지수) 정수 결과를 내고 역양자화(dequantization) 유닛이 이를 FP22로 한 번에 변환한다 `[0145]`. 최적화된 지수 처리 경로는 32×32 예시 어레이 기준 64×64개의 하프/풀 애더를 절약한다 `[0183]`. 이중 비트셀 설계는 다음 가중치 행렬을 로드하는 동안 기존 가중치로 연산을 계속하게 하며 `[0154]`, 일부 실시예에서는 100% 가동률도 가능하다고 적는다 `[0154]`.

**근거 (verbatim):**
- `[0131]`: "Reduced precision formats (i.e., reduced bit widths) can greatly increase computation speed and lower memory usage with minimal loss in accuracy, and thereby greatly improve the latency and throughput of a deep learning process in a neural network."
  (저정밀 형식이 정확도 손실을 최소화하며 연산 속도와 메모리 사용, 지연과 처리량을 크게 개선한다는 뜻)
- `[0154]`: "allowing concurrent loading of the next matrix of weight values while the previous weight values are still in use for the VMM computation"
  (이전 가중치가 아직 연산에 쓰이는 동안 다음 가중치 행렬을 동시에 로드할 수 있다는 뜻)

## ⑤ 회사 프로모션 글/기술과의 연결

**특허가 실제로 뒷받침하는 것**
- OpenAI가 자사 명의로, 트랜지스터/래치 수준까지 내려가는 AI 연산 회로 설계를 출원했다는 사실 자체. 문서에는 회로도 수준의 비트셀 설계 `[0227]`, 클록 타이밍 조건 `[0234]`, 테스트용 스캔 체인 `[0118]` 까지 담겨 있다.
- 가출원 우선일은 2024-11-22로, OpenAI-Broadcom 협력 발표(2025-10-13)보다 약 11개월 앞선다. 즉 회사가 "OpenAI가 설계한 가속기"를 공표하기 훨씬 전부터 회로 수준 설계 기록이 존재한다.

**회사 주장**
- OpenAI와 Broadcom은 2025-10-13에 10기가와트(GW) 규모의 "OpenAI-designed AI accelerators" 배치 협력을 발표했고, 랙 배치는 2026년 하반기 시작, 2029년 말 완료 목표다 (fact-check-log: openai-broadcom-10gw, company-claimed).
- 2026-06-24 첫 커스텀 칩 "Jalapeño"(LLM 추론 최적화)를 공개했고 2026년 말 초기 배치를 목표로 한다 (openai-broadcom-jalapeno, company-claimed).
- 발명자 Jean-Didier Allegrucci는 Apple에서 약 17년간 SoC 엔지니어링을 이끈 인물로 2024년 6월 Rain AI(인메모리 컴퓨트 스타트업)에 합류했다고 보도되었다 (allegrucci-apple-rain, press-reported). Rain AI 매각 과정에서 OpenAI가 인력을 흡수했다는 보도가 있으나 인수 완료는 확인되지 않았다 (rain-ai-sale-openai-talks, press-reported; 단정 금지).
- 이 특허가 Jalapeño 또는 Broadcom 공동 개발 칩에 실제로 들어가는지는 어떤 자료도 말해 주지 않는다. 연결은 "방향의 증거"까지만 가능하다.

## ⑥ 아는 것/모르는 것 경계 지도

**(a) 이 런의 증거가 확립한 것**
- 공개 공보 전문 확인: 출원인/양수인 OpenAI Opco, LLC, 발명자 5인, 출원일 2025-11-21, 가출원 2024-11-22, 공개일 2026-05-28.
- 청구항 1, 20, 29의 문언(가수 정렬 + 정수 덧셈기 트리)과 청구항 11-19 취소 표기.
- OpenAI-Broadcom 10GW 발표(2025-10-13)와 Jalapeño 공개(2026-06-24)는 회사 공식 자료로 확인.
- OCP MX(마이크로스케일링) 블록 형식 표준이 2023년 9월 v1.0으로 공개된 사실 (AMD/Arm/Intel/Meta/Microsoft/NVIDIA/Qualcomm 참여).

**(b) 증거 밖에 있는 것 - 단정 불가 리스트**
- 이 출원의 심사 경과(거절·보정 여부)와 최종 등록 여부. 청구항은 심사 중 바뀔 수 있다.
- 취소된 청구항 11-19의 내용이 어디로 갔는지(분할·계속출원 여부는 미확인).
- 가출원(2024-11-22) 시점에 발명자들이 어느 조직 소속이었는지. Rain AI 인수 완료 여부.
- Jalapeño 또는 Broadcom 공동 칩이 이 출원의 청구항을 실시하는지 여부.
- 이 설계가 실리콘으로 제조되었는지, 성능 수치가 실측인지 여부(명세서 수치는 설계 예시다).

**(c) 원문 자체가 열어둔 것**
- 명세서는 SRAM 외에 NAND/NOR 플래시, ReRAM 등 다른 메모리, 그리고 디지털이 아닌 아날로그 CIM 구현도 열어둔다 `[0133]`.
- 형식 조합, 블록 크기, 비트셀 수(최소공배수 방식 `[0190]`), 시프터 구현(디코딩 방식 `[0204]` 대 로그 트리 `[0206]`) 등은 "may/can" 표현의 대안 실시예로 남겨져 있다.
- 메모리 셀당 비트셀 수도 1개, 2개 또는 그 이상으로 열려 있다 `[0162]`.
